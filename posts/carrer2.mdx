> 이전 **레거시 VB 시스템 웹 전환 프로젝트(1차 마이그레이션)** 에서 만든 Web + API 구조 위에  
> **전자결재 시스템**과 **프로젝트 MD 산정 시스템**을 추가로 개발하며,  
> 기존 서비스를 **유지보수하면서 동시에 기능을 확장·고도화**한 프로젝트입니다.

# 1. 개요

### 1. 일정

2025년 6월 ~ 2025년 9월

### 2. 프로젝트 요약

- 1차 마이그레이션에서 구축한 **Web + API + DB 계층 구조를 유지**하면서, 새로운 도메인(전자결재, MD 산정)을 추가
- 결재 단계·권한 체계를 반영한 **전자결재 Workflow 시스템** 구축
- 프로젝트별 투입 공수를 계산하는 **MD 산정 로직을 시스템화**
- 기존 수작업·엑셀 기반으로 관리되던 프로세스를 DB·Stored Procedure로 정형화

### 3. 나의 역할

- 기존 마이그레이션 시스템 유지보수
  - 이전 프로젝트에서 구축한 API/DB 구조를 분석해, **호환성을 깨지 않으면서** 신규 기능을 붙일 수 있는 확장 포인트 정리
  - 공통 모듈(인증, 권한, 공통 코드 등)을 재사용·리팩터링하여, 중복 구현 없이 전자결재/MD 기능에서 활용
- 전자결재 시스템
  - 결재 단계/상태 전이를 정의하고, **Workflow 모델 및 상태 관리 로직 설계**
  - 결재 권한 검증, 접근 제어 로직 구현
  - Stored Procedure 기반 결재 처리 트랜잭션 설계·구현
- MD 산정 시스템
  - 기존 산정 기준·수작업 프로세스를 분석하여 **T-SQL Stored Procedure로 계산 로직 구현**
  - 다중 테이블 Join 기반 **집계/조회 쿼리** 작성
  - 실적 관리 및 조회용 웹 화면 구현
- 보안 점검 대응
  - 사내 보안 점검에서 지적된 입력 검증·권한 관리 관련 이슈를 분석하고, 공통 미들웨어 레벨에서 검증·로그 로직을 보완
  - 전자결재/MD 기능에서 발생할 수 있는 민감 데이터 접근 시나리오를 점검해, **권한 체크·감사 로그를 강화**하는 방향으로 개선

### 4. 느낀점/배운점

- “사내 규칙”을 코드로 옮길 때, **정책·예외 케이스를 글로 먼저 정리하는 과정이 중요**하다는 것을 다시 한 번 느낌.  
  전자결재/MD 산정처럼 이해관계자가 많은 도메인은, 규칙을 코드로 바로 옮기기보다  
  문서·표 형태로 합의한 뒤 구현해야 나중에 유지보수나 기준 변경이 들어왔을 때 근거를 설명하기가 훨씬 수월했음.
- Stored Procedure를 적극적으로 활용하면서,  
  **DB 레이어에서 처리하는 것이 더 적합한 로직과 아닌 로직을 구분**하는 기준을 잡게 됨.  
  조회·집계·정합성 보장이 중요한 로직은 DB에 두고, 화면·권한·Workflow 같은 부분은 애플리케이션 레이어에서 처리하는 식으로  
  “어디까지 DB에 맡기고, 어디서부터는 서비스 레이어에서 책임져야 하는지”에 대한 감각이 생겼음.
- 특히 기존 마이그레이션 시스템 위에 기능을 얹는 작업을 통해,  
  **처음 설계한 아키텍처가 이후 확장·유지보수에 어떤 영향을 주는지**를 직접 체감할 수 있었음.  
  초기에 공통 모듈·레이어 분리를 잘 해둔 덕분에 전자결재/MD 기능을 비교적 안전하게 추가할 수 있었지만,  
  반대로 구조가 애매하게 설계된 부분은 이번 고도화 단계에서 손을 더 많이 봐야 했고,  
  이 경험을 바탕으로 이후 프로젝트에서는 “나중에 붙을 기능”까지 고려한 확장 포인트 설계에 더 신경 쓰게 되었음.

### 5. 개발 시 발생한 문제점 / 해결책

#### 5.1.1 문제점

- 전자결재/MD 산정 규칙이 여러 부서·담당자에 걸쳐 있어,  
  개발이 진행되는 중에도 **담당자 요구사항·정책 변경·예외 케이스**가 계속 추가되거나 바뀌는 일이 잦았음.

#### 5.1.2 해결책

- 요구사항·규칙·예외 사항을 먼저 문서로 정리하고,  
  변경 가능성이 높은 항목은 분리해 코드 수정 없이 조정할 수 있게 설계

#### 5.2.1 문제점

- 보안 점검 지적사항 중 일부는 **기존 공통 모듈 설계의 한계를 드러내는 이슈**가 있었고,  
  단순 패치로는 재발 방지를 보장하기 어려운 부분이 있었음.

#### 5.2.2 해결책

- 입력 검증·권한 체크·감사 로그를 담당하는 공통 미들웨어를 재검토하고,  
  전자결재/MD 기능에서 발견된 케이스를 반영해 **공통 모듈 자체를 개선**하여 다른 기능에도 동일하게 적용
- 보안 점검 결과를 정리해, 이후 신규 기능 개발 시 체크해야 할 **보안 체크리스트**로 재활용할 수 있도록 문서화함.

## 3. 회고 & 다음 개선

- 1차 마이그레이션에서 만든 구조 위에 전자결재·MD 기능을 올리면서,  
  **“운영 중인 시스템을 멈추지 않고 고도화하는 일”** 이 신규 개발과는 다른 어려움이라는 것을 체감했음.  
  배포·반영 과정을 안정적으로 하려면 **자동 배포(CI/CD) 체계**를 갖추는 것이 필요하다고 느꼈음.
- 요구사항이 계속 바뀌는 환경에서는, 코드만 손보는 것보다  
  **정책·규칙·예외를 문서로 정리하고 당담자와 합의하는 과정**이 먼저 선행되어야 개발 효율이 오른다는 걸 배움.
- 다음에는 초기 설계 단계에서부터 **설정 기반 구조(코드 테이블, 환경 설정)** 를 더 적극적으로 도입해,  
  정책·담당자 변경에 코드 수정 없이 대응할 수 있는 범위를 넓히고자 함.
- 보안 점검에서 드러난 이슈들을 계기로,  
  신규 기능을 설계할 때부터 **입력 검증·권한·로그를 체크리스트로 두고 개발**하는 습관을 들이려 함.
