- PM2 · Nginx · Cloudflare Tunnel · 배포 스크립트를 이용한 **배포/운영 환경 구성**

### 4. 개발시 발생한 문제점/해결책

#### 4.1.1 문제점

- 기능이 늘수록 “공통 규칙(다국어/권한/응답 포맷)”이 깨지기 쉽다
- API가 늘어나면 각 라우트마다 언어 파라미터 처리 방식이 달라지고, 에러 응답 포맷도 제각각이 되기 쉬움.
- 특히 “게임 도구”는 신규 기능이 빠르게 추가되는 편이라 **일관성 유지 비용**이 늘어남.

#### 4.1.2 해결책

- **미들웨어에서 `lang` 쿠키를 검증**하고, API 요청에 `x-lang` 헤더로 주입해 모든 라우트에서 동일하게 사용함.
- **공통 API 응답 유틸**을 만들어 `success/data` 또는 `success/message` 형태로 통일함.
- 프로덕션에서는 내부 오류 메시지를 숨기고(고정 메시지), 실제 메시지는 서버 로그로만 남겨 **정보 노출을 방지**처리함.

#### 4.2.1 문제점

- “누가 작성했는지” 식별 없이 권한을 부여하기 어려운데, 로그인 기능까지 도입하면, 복잡도가 커지고, 사용 장벽이 증가함

#### 4.2.2 해결책

- `POST /api/uid`로 **서버에서 UUID를 발급**하고(브라우저 쿠키/스토리지에서 사용), UID 기반으로 기능별 권한을 설계할 수 있게 함.
- UID 조회 시 `touch(updated_at)`로 “활동성”도 함께 관리할 수 있게 구성함.

## 2. 핵심 기능

- **심법 뽑기(시뮬레이터)**: 게임 내 확률/규칙을 웹에서 재현해 빠른 시뮬레이션
- **스무고개**: 족보/검색을 위한 데이터 제공
- **만사록**: 지역/세부지역 필터 기반 목록 조회 및 상세 제공
- **다국어(ko/en/ja/zh)**: UI + API 공통 적용
- **API 문서화**: Swagger 주석 기반으로 `/api-doc`에서 자동 노출

## 4. 설계/구현 포인트

### 4.1 레이어드 아키텍처로 변경 비용을 줄임

- **API Routes**(`app/api`): HTTP/쿼리/쿠키 처리 + 응답 매핑
- **Service**(`service`): 비즈니스 로직, 검증, 조합
- **Repository**(`repo`): DB 접근 쿼리 책임 분리
- **Types**(`types`): 도메인/DTO 타입으로 안정성 확보

### 4.2 다국어를 “기능”이 아니라 “인프라”로 만들기

- 지원 언어: `ko/en/ja/zh`
- 쿠키 `lang` → (미들웨어 검증) → `x-lang` 헤더 → API에서 `getLangFromRequest()`로 사용
- 라우트마다 언어 처리 코드가 복제되지 않도록 중앙화

### 4.3 운영 관점의 에러 처리

- 개발: 원인 파악이 쉽도록 실제 메시지 반환
- 프로덕션: 고정 메시지로 응답하고 서버 로그에만 상세를 남김
