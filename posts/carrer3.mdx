> LLM 기반의 **도메인 특화 검색 시스템**을 구현하기 위해,  
> Hugging Face 기반 임베딩 모델과 LangChain을 활용한 검색 파이프라인을 설계했고,  
> 이 생태계를 가장 잘 활용할 수 있는 Python을 사용하여 API 서비스를 구축한 프로젝트입니다.

# 1. 개요

### 1. 일정

2025년 10월 ~ 2025년 12월

### 2. 프로젝트 요약

- Python으로 작성된 데이터 처리 로직을 분석해 **FastAPI 기반 API 서비스**로 재구성
- Hugging Face 임베딩 모델과 LangChain을 활용해, **LLM 기반 검색·질의응답 API**를 설계·구현
- 정기 실행이 필요한 작업을 스케줄러에 올려 **자동 처리 파이프라인** 구축
- 처리 결과를 DB에 저장하고, 기존 서비스와 **API 연동**으로 기능 통합

### 3. 나의 역할

- 데이터 처리 로직 분석 및 **API 설계**
  - 입력/출력/에러 케이스를 정리하고, 기존 서비스와의 **API 계약 정의**
- FastAPI 기반 **엔드포인트 구현 및 비즈니스 로직 이관**
  - 스크립트 단위 로직을 서비스 레이어로 옮기고, 에러 처리·재시도·로깅 구조 설계
- 정기 배치를 위한 **스케줄러 구성** 및 실행 상태·모니터링 구조 설계
- 처리 결과를 MSSQL에 저장하는 **파이프라인 및 테이블 설계**
- 기존 서비스와의 연동 테스트 및 **API 버전·호환성** 관리

### 4. 느낀점/배운점

- “스크립트 한 번 돌리는 것”과 **“서비스로 상시 운영하는 것”의 책임 범위 차이**를 체감함.  
  에러 처리, 재시도, 모니터링, 배포 전략 등을 서비스 수준으로 설계해야 안정적으로 돌아간다는 걸 배움.
- 데이터 처리 작업을 API·배치로 서비스화하면서,  
  **로그·메트릭·에러 핸들링을 초기에 잘 설계하는 것이 운영 안정성에 중요**하다는 것을 느낌.
- 기존 시스템과의 **API 계약을 명확히 하고, 입력/출력/예외를 문서화**해 두면  
  연동 이슈와 유지보수 비용을 크게 줄일 수 있다는 경험을 함.

### 5. 개발 시 발생한 문제점 / 해결책

#### 5.1.1 문제점

- 기존 Python 스크립트는 “한 번 실행하고 끝” 가정이라,  
  **에러 발생 시 재시도·부분 실패·로깅** 등 서비스 운영에 필요한 처리가 없었음.

#### 5.1.2 해결책

- 처리 단계별로 **try/except·로깅·재시도 정책**을 넣고,  
  실패한 구간만 재실행할 수 있도록 구간 단위 설계를 적용
- 스케줄러·API 응답에 **실행 결과·에러 코드**를 남기고, 모니터링·알림과 연동할 수 있게 구성

#### 5.2.1 문제점

- 정기 배치 실행 시간과 기존 서비스의 **피크 타임이 겹치거나**,  
  처리량이 많을 때 **리소스·DB 부하**가 예상보다 커지는 경우가 있었음.

#### 5.2.2 해결책

- 배치 실행 시각을 서비스 사용 패턴을 고려해 조정하고,  
  대량 처리 구간은 **청크 단위 처리·타임아웃**을 두어 한 번에 부하가 몰리지 않도록 설계
- 필요 시 **큐·비동기 처리** 도입을 검토하고, 우선순위가 높은 API와 배치가 서로 방해하지 않도록 리소스 사용 기준을 정리

#### 5.3.1 문제점

- 기존 서비스와의 **API 연동 시 스키마·버전 차이**로 인해,  
  데이터 형식이나 필드 변경 시 양쪽을 동시에 수정해야 하는 상황이 발생할 수 있었음.

#### 5.3.2 해결책

- 요청/응답 스키마를 문서화하고, **버전을 포함한 API 경로 또는 헤더**로 관리해  
  기존 클라이언트와의 호환성을 유지하면서 변경을 단계적으로 적용
- 연동 테스트 시나리오와 **계약(contract) 검증**을 자동화해,  
  스키마 변경 시 영향 범위를 빠르게 확인할 수 있도록 함.

## 3. 회고 & 다음 개선

- 스크립트를 서비스로 옮기는 과정에서 **에러 처리·재시도·로깅·모니터링**을 처음부터 설계해야  
  나중에 운영 부담이 줄어든다는 걸 다시 한 번 체감했음.
- 다음에는 **배치·API 성능·리소스 사용**을 초기 단계에서부터 가정하고,  
  부하 테스트·모니터링 구간을 같이 설계해 두고 싶음.
- API 연동 시 **계약·버전 관리**를 명확히 해 두면 변경 비용이 줄어들므로,  
  이후 프로젝트에서도 API 스펙 문서화와 자동 검증을 습관화하려 함.
