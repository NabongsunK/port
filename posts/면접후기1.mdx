최근에 면접을 보고왔다.

신입으로 들어갔을때에도 면접에 대한 걱정이 컸었고, 지금도 그때랑 별반 다르지 않다.

지금은 그저 누가 나한테 vue코드 뭉탱이로 줘도 놀라지 않을 것이고,

SP 5000줄짜리를 봐도 흐음 이정도면 0.5계획 수준이구만
(내가 전 회사에서 봤었던 SP 중 제일 긴 것이 만줄 수준이었다.)

이정도인데, 막상 LEFT JOIN에 대해서 설명해 보라 했을 때 어버버했던 걸 보면 부끄럽다.

그때 어떻게 대처했어야 했는지 하나쯤 적어두면 좋을 것 같아서 피드백을 해보자.

내가 했던 실수들

1. 들어가자마자 면접관이 8명 있어서 당황했다.
   사실 그 업무에 대해 큰 기대를 안 했어서, 전 회사처럼 대표 혼자서 하는 걸 기대했다.
   (하긴 그래서 그때도 거기만 붙었었지)
   그런데 실제 드라마에서 볼 만한 사람들이 앉아있었고, 그때 아무 생각이 들지 않았다.

그래서 바로 앉아버렸다.
면접관분들의 벙쩌하는 표정이 보였다.

다음에는 들어가서 인사하고, 그다음 말을 기다리도록 하자.
정상적인 회사라면 "앉으세요" 라고 정도는 말해주겠지.

2. 첫번째 질문 최근에 본 책이 있나요?
   사실 책을 안 읽은 지가 8년이 지났다. 당황해서 군대 있을 때 읽었던 마션에 대한 이야기를 했었다.
   첫 질문이라면 내가 어떤 사람인지 알고 싶어할 텐데, 이럴 때는 사실 아직 잘 모르겠다.

그냥 최근엔 책을 안 읽었었고, 기억나는 거는 최근에 AI 에이전트 관련해서 관심이 깊어졌는데,
이때 오픈마누스라는 오픈소스 AI 에이전트가 유명하다고 해서, 그것에 대한 프로젝트 README 문서를 보면서
어떤 식으로 만들어야 할지 생각을 많이 했었다.

이정도로 답변하면 좋았을 것 같다.

3. 회사에서 이룬 제일 큰 성과가 무엇인 거 같으세요??
   사람이 많이 바뀐 상황에서 우당탕탕 해서라도 프로젝트가 진행되었었는데,
   마무리 단계에서 그쪽 담당자가 고생했다 했던 일이 떠오른다 라고 답변했었다.
   이정도 성과면 대단한 건가?
   우당탕탕 회사여서 마이너스가 있었으려나 싶긴 하다.

근데 진짜 큰 성과가 저건데
(실제로 상당히 우당탕탕 하기도 했고)
아니면 JOIN 기가 막히게 걸어서 SP만으로 기능 구현한 거?
그걸 말했어야 했나

4. JAVA 안 해봤는데 ㄱㅊ?
   이 질문은 사실 예상질문에 있었다.
   그래서 예상답변처럼 옛날에 전공에서 JAVA 자체는 다뤘었고,
   API 구조만 잘 지키면 언어는 크게 문제 없을 것 같다라고 답변하였다.

5. SP 썼다고 했는데 왜 쓸까요?
   나는 네트워크 이슈가 적을 것 같다 라고만 답했었다.
   사실 속마음은 그냥 거기가 썼으니까 썼지 왜 쓰긴 해.. 이 마인드였다.
   지금 다시 생각해보면 그때도 동료들이랑 왜 SP로 할까요? 라고 이야기를 했었는데,
   지금 다시 물어보면 단점을 먼저 이야기하는 게 좋을 수도 있을 것 같다.

도메인 흐름이 밖에서 보이지 않는 단점이 있다고 들었다.
아마 실무적으로는 그 사람들이 도메인 흐름을 DB 내부에서 관리하고 싶어했을 것 같다.
그리고 데이터 갯수도 몇만개, 몇십만개 수준으로 관리하다 보니
네트워크 이슈에서 자유로울 수 있을 것 같다.

이정도로 이야기하는 게 좋지 않겠는가?!

6. LEFT JOIN에 대해서 알고 있나?
   진짜로 내가 짜왔던 쿼리 대부분이 LEFT JOIN이었는데
   갑자기 어? 이걸 왜 물어보지?
   INNER JOIN이랑 차이 설명해야 하나?
   아닌데 그럼 JOIN을 언제 쓰는지 말해야 하나?
   교집합도 아니고, 차집합도 아니고 왼쪽 거를 오른쪽 거에 붙이는 건데
   이렇게 말해도 되는 걸까?
   어버버하다가 이야기를 못했다.

일단 사전적 정의를 이야기하는 것이 좋을 것 같다.
LEFT JOIN은 왼쪽 테이블의 모든 행을 기준으로 가져오고, 오른쪽에 매칭이 없으면 NULL로 채우는 조인입니다.
그래서 "기준 데이터는 유지하면서 부가 정보를 붙일 때" 사용합니다.

실무에서는 정규화된 테이블에서 기준 엔터티를 유지한 채 관련 정보를 붙여 조회할 때 LEFT JOIN을 자주 사용했습니다.
또한 정규화가 안 되어 있는 테이블이 있었는데, 정규화를 하면서 원래 조회 형태를 뷰로 만드는 작업도 했었다.
이때 사용했었다.

이정도로 답하면 좋은 답변이 되었을 것 같다.
