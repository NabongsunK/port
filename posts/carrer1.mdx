> **VB.NET WinForms 기반 레거시 내부 시스템**을  
> **웹 기반 SPA + REST API 아키텍처**로 전환한 프로젝트.  
> 화면만 바꾸는 수준이 아니라, **업무 로직·DB 연동 방식까지 재설계**한 마이그레이션 작업.

# 1. 개요

### 1. 일정

2024년 4월 ~ 2025년 4월

### 2. 프로젝트 요약

- 기존 **VB.NET WinForms**로 작성된 사내 업무 시스템을 **웹 기반 시스템**으로 전환
- 화면·이벤트 중심 로직을 **SPA 구조(React/Vue) + REST API + DB 계층**으로 분리
- 레거시 코드에 얽혀 있던 **DB 연동/비즈니스 로직을 정리·모듈화**하여 유지보수성 향상

---

### 3. 나의 역할

- VB.NET WinForms 기반 레거시 시스템 분석후, 필요한 프로시저 및 뷰 정리
- 해당 프로시저를 호출하는 Golang API 서버 구현
- 기존 쿼리/데이터 접근 코드를 계층형 구조(Controller/Service/Repository) 로 이동
- VB.NET WinForms 기반 레거시 시스템의 이벤트 기반 로직을 분리하여 처리

---

### 4. 느낀점/배운점

- 화면과 로직, 데이터 계층이 섞여 있는 레거시 구조를 분리하는 과정에서  
  **“어디까지 그대로 가져가고, 어디부터 새로 설계할지” 경계선을 잡는 게 중요**하다는 걸 느낌.
- 하나의 거대한 폼 이벤트에 묶여 있는 처리를 잘라내면서  
  **도메인별로 책임을 나누고, API 기준으로 다시 생각하는 연습**을 많이 하게 됨.
- 사용자는 **새 UI**보다 **기존과 동일하게 동작하는가**를 더 중요하게 보기 때문에,  
  UX 개선과 기능 파리티 사이의 밸런스를 맞추는 경험을 함.
- 협업 및 의사소통 능력 향상
  - 동료 개발자와 레거시 화면을 같이 보면서 요구사항을 다시 정의하고,  
    우선순위를 합의해 가는 과정에서 **요구사항 정리·조율 능력**이 자연스럽게 향상되었음.
- 개발 시 발생한 문제점/해결책 처리 능력 향상
  - 레거시 분석 → 설계 → 구현 → 검증 과정에서 계속해서 이슈가 생겼고,  
    각 문제를 재현·원인 분석·대안 제시까지 연결하는 **문제 해결 루틴**을 체득했음.
- 레거시 시스템을 웹 기반 시스템으로 전환하는 경험 향상
  - 단순 신규 개발이 아니라, **기존 시스템을 이해하고 안전하게 옮기는 경험**을 통해  
    추후 다른 레거시 마이그레이션에서도 재사용할 수 있는 기준과 노하우를 쌓았음.

---

### 5. 개발 시 발생한 문제점 / 해결책

#### 5.1.1 문제점

- 폼/이벤트 안에 비즈니스 로직 + 화면 제어 + 데이터 접근이 섞여 있어,  
  그대로 옮기면 새 프로젝트도 금방 레거시가 될 위험이 있었음.

#### 5.1.2 해결책

- 화면 단에서 필요한 것을 **입·출력 관점(입력 필드, 검증, 저장 대상)** 으로 먼저 정리
- 그 뒤, 비즈니스 규칙을 **Service 계층**으로 빼고,  
  DB 접근은 **Repository 계층**에서만 처리하도록 책임을 나눔.

#### 5.2.1 문제점

- 프로젝트 팀원이 자주 바뀌어 업무 맥락과 기술 결정 이유가 유실되고, 인수인계 비용이 반복적으로 발생했음

#### 5.2.2 해결책

- 레거시 분석 결과, API 명세, 배포/실행 방법을 온보딩 문서·Runbook 으로 정리해 인수인계 시간을 단축
- Controller/Service/Repository 계층 구조와 네이밍 규칙을 통일해, 새로 합류한 팀원도 같은 패턴으로 바로 기여할 수 있게 함

#### 5.3.1 문제점

- 초기 일정 산정 시 레거시 분석·테스트·사용자 검증에 필요한 시간을 과소평가해,  
  마이그레이션 막바지에 일정 압박이 집중되는 상황이 발생함

#### 5.3.2 해결책

- 필수 기능/부가 기능을 나눠, 마감이 촉박할 때는 필수 기능 안정화에 집중하고 부가 기능은 후속 릴리스로 조정

## 2. 사용된 기술

- **Frontend**
  - React / Vue: SPA 기반 화면 구성
- **Backend**
  - Golang: REST API 서버
- **Database**
  - MSSQL: 기존 레거시 DB 유지, 저장 프로시저·뷰 정리
  - MongoDB: 일부 로그/비정형 데이터 저장
- **기타**
  - Git

## 3. 회고 & 다음 개선

- 이번 프로젝트를 통해 **레거시를 그대로 옮기는 것과, 업무를 다시 설계하는 것의 차이**를 몸으로 익히게 되었음.  
  단순 코드 변환이 아니라, 도메인·API·데이터 흐름을 기준으로 다시 바라보는 시각을 키웠다는 점이 가장 큰 수확.
- 다음에는 초기 단계에서 **도메인 모델링과 API 설계를 더 체계적으로 문서화**하고,  
  레거시 화면·기능을 그 관점에서 매핑하는 워크숍을 먼저 진행해, 전체 구조를 팀과 더 빨리 공유하고 싶음.
- 일정·인력 변동에 대비해, **온보딩 문서·코딩 규칙·폴더 구조**를 프로젝트 초기에 잡아두고 시작하면  
  중간에 팀 구성이 바뀌더라도 속도와 품질을 더 안정적으로 유지할 수 있을 것이라 느낌.
- 일정 조율과 우선순위 결정 과정에서, **담당자과의 지속적인 소통이 일정 리스크를 줄이는 데 핵심**이라는 점을 다시 한 번 느꼈음.
- 유닛 테스트·자동화된 테스트 환경이 충분히 갖춰져 있지 않아 배포 직전에 문제가 발견되는 경우가 있었고,  
  다음에는 **주요 도메인 로직에 대한 유닛 테스트와 최소한의 회귀 테스트 환경을 초기에 같이 설계**해야겠다고 느꼈음.
