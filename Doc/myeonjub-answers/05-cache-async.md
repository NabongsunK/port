# 5. 캐시·비동기 처리

## 인메모리 캐시(Redis 등)

**답변 요지:**  
자주 읽고 덜 바뀌는 데이터를 메모리에 두어 DB·외부 API 부하를 줄입니다. **키 설계**는 충돌 없이 의미가 드러나게, **TTL**로 오래된 값을 자동 제거합니다. **캐시 무효화**는 데이터 변경 시 키 삭제·버전 접두사·pub/sub 등 전략 중 하나를 택합니다.

---

## 세션 스토어·분산 락(Redis)

**답변 요지:**  
세션을 앱 서버 메모리가 아니라 Redis에 두면 **스케일아웃** 시에도 공유됩니다. **분산 락**은 `SET NX`+TTL, Redlock 논쟁은 알고 있으면 언급. 짧은 임계 구간에서는 멱등과 함께 쓰는 편이 안전합니다.

---

## 메시지 큐(Kafka, RabbitMQ, SQS 등)

**답변 요지:**  
요청 스레드와 무거운 작업을 분리해 **응답 시간**을 줄이고, **피크 흡수·재시도·다운스트림 장애 격리**에 씁니다. **최소 한 번(at-least-once)** 전달은 **소비 측 멱등**과 짝지어 설계합니다.

---

## 웹소켓 vs SSE vs 폴링

**답변 요지:**

- **폴링**: 구현 단순, 지연·부하 트레이드오프.
- **SSE**: HTTP 단방향 스트림, 서버→클라이언트 푸시·재연결에 적합한 경우 많음.
- **웹소켓**: 양방향, 채팅·협업 편집 등. 인프라·스케일(스티키 세션·메시지 브로커)을 함께 고려해야 합니다.
