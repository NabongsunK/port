# 3. API 설계·보안

## RESTful API 설계, 리소스 네이밍·버전

**답변 요지:**  
URL은 무엇을 다루는지 명사형으로 표현하고, 행동은 HTTP메서드로 구분합니다.
만약에 1번사용자를 조회해야하면, /users/1 이런식으로하고
유저정보를 수정하는건 /users 로 PUT/PATCH를 사용하고
회원가입 같은거는 /users로 POST
회원삭제는 /users로 DELETE 처리할꺼 같습니다.

---

## 인증 vs 인가

**답변 요지:**

- **인증(Authentication)**: “이 사람이 누구인지” 확인(로그인).
- **인가(Authorization)**: “이 작업을 할 권한이 있는지” 확인(역할·스코프).  
  JWT만 검증했다고 끝이 아니라, **리소스 단위 권한**(본인 데이터만 수정 등)을 체크해야 합니다.

**경험 예시(전자결재):**

- **인증**은 먼저 로그인으로 **누가 접속했는지**(관리자/피관리자 등) 식별하는 단계입니다.
- **인가**는 그 다음입니다. **역할에 따라** 목록·검색 **조회 범위가 달라지게** 했습니다—피관리자는 **본인이 올린 결재** 위주로 보고, 관리자는 **결재 처리 대상**이 되는 건을 **더 넓게** 볼 수 있게 **서버에서 조회 조건을 분기**했습니다.
- **승인·반려**는 “로그인만 했다”로 되는 게 아니라, **그 문서의 현재 단계에서 결재 권한이 있는 역할**인지까지 **리소스 단위로** 검증했습니다. 즉 **워크플로 상태 전이**와 **권한 체크**를 같이 봤습니다.

## JWT 구조·만료·갱신·저장 주의

**답변 요지:**  
헤더·페이로드·서명으로 구성되고, 페이로드는 **Base64 디코딩만 하면 읽힘**(비밀 넣지 않기).  
만료(`exp`)를 짧게 두고, 리프레시 토큰으로 재발급하거나 서버 세션과 혼합할 수 있습니다.  
클라이언트 저장 시 XSS에 민감하므로 HttpOnly 쿠키 vs localStorage 트레이드오프를 인지합니다.

---

## CSRF, XSS, SQL Injection — 서버에서 할 수 있는 것

**답변 요지:**

- **CSRF**: 쿠키 기반 인증 시 `SameSite`, CSRF 토큰, 중요한 작업은 POST+검증.
- **XSS**: 입력 검증·출력 이스케이프, CSP 헤더, 쿠키 HttpOnly.
- **SQL Injection**: **파라미터 바인딩(Prepared Statement)**, ORM 올바른 사용, 동적 쿼리 최소화.

## 비밀번호 해시·솔트, 시크릿 관리

**답변 요지:**  
비밀번호는 평문 저장 금지, **bcrypt·Argon2** 등 적응형 해시 + **솔트(요즘은 라이브러리가 자동)**.  
API 키·DB 비밀번호는 코드에 넣지 않고 **환경변수·시크릿 매니저**, 저장소에는 `.env` 미커밋.

**POST 할 때 클라이언트에서 해시해서 보내나?**

- **일반적인 웹·모바일 로그인·회원가입**에서는 **아니요**. 요청 본문에는 **사용자가 입력한 비밀번호(평문)** 를 넣고, **HTTPS(TLS)** 로 전송 구간을 암호화합니다. 서버가 받은 뒤 **bcrypt/Argon2로 해시**해 DB에는 **해시만** 저장하고, 로그인 시에도 서버가 같은 방식으로 **저장된 해시와 비교**합니다.
- **이유:** 해시는 “한 번 만들면 원문을 알 수 없게 하는 값”이지, 네트워크상의 **두 번째 비밀번호**가 아닙니다. 클라이언트에서 단순 해시만 해서 보내면, 그 값이 **그대로 재전송(replay) 공격에 쓰일 수 있고**, 서버 입장에서는 **라운드·솔트 정책을 일관되게 적용**하기도 어렵습니다. 그래서 **전송은 TLS**, **저장·검증은 서버에서 적응형 해시**가 표준 패턴입니다.
- **예외/고급:** SRP 같은 프로토콜은 서버가 평문을 갖지 않게 설계할 수 있지만, 구현·운영 부담이 커서 면접·일반 서비스 답변에서는 **HTTPS + 서버 해시**로 정리하는 경우가 많습니다.

**한 줄로:** 전송 구간은 **HTTPS**, DB에는 **해시만** — POST 본문의 “해시된 비밀번호”가 아니라 **TLS 안에서 보호되는 평문 입력**을 서버가 받아 처리한다고 보면 됩니다.
