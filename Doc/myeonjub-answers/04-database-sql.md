# 4. 데이터베이스·SQL

## 4.1 RDBMS vs NoSQL, 정규화·반정규화, PK·FK·CASCADE

**Q. RDBMS랑 NoSQL 차이를 어떻게 보시나요?**

**A.** RDBMS는 스키마가 비교적 고정되고 ACID 트랜잭션·JOIN에 강해서, 트랜잭션이 중요한 업무나 복잡한 조회·보고서에 잘 맞는 편입니다. NoSQL은 문서·키값·컬럼·그래프처럼 모델마다 강점이 다르고, 스키마가 유연하고 수평 확장에 유리한 경우가 많은데, 일관성 모델은 제품마다 달라서 도입할 때 그 부분까지 같이 봐야 합니다.

**Q. 정규화랑 반정규화는 언제 쓰나요?**

**A.** 정규화는 중복을 줄여서 무결성을 지키고 변경을 쉽게 만드는 쪽이고, 반정규화는 읽기 성능이나 쿼리 단순화를 위해 의도적으로 중복을 두는 겁니다. 대신 갱신 비용이 늘고, 일관성을 애플리케이션이나 운영에서 더 챙겨야 하는 트레이드오프가 있습니다.

**Q. PK, FK, CASCADE는 어떻게 이해하시나요?**

**A.** PK는 행을 유일하게 식별하고, FK는 다른 테이블을 참조하면서 참조 무결성을 맞추는 역할입니다. `ON DELETE CASCADE` 같은 옵션은 부모 행을 지울 때 자식을 DB가 어떻게 처리할지 정책을 DB에 맡기는 거라서, 운영·데이터 영향을 알고 써야 합니다.

## 4.2 JOIN, 인덱스, N+1, EXPLAIN

**Q. INNER JOIN이랑 LEFT JOIN 차이는요?**

**A.** INNER JOIN은 양쪽에 매칭되는 행만 나오고, LEFT JOIN은 왼쪽 테이블은 전부 나오고 오른쪽은 매칭되는 것만 붙습니다. 쿼리 짤 때는 어떤 테이블이 드라이빙인지, 필터가 먼저 걸리는지까지 같이 봅니다.

**Q. 인덱스는 왜 쓰고, 설계할 때 뭐가 중요한가요?**

**A.** 조회 조건이나 정렬에 쓰는 컬럼을 B-Tree 같은 구조로 빠르게 찾기 위해서입니다. 인덱스가 많으면 쓰기랑 저장 공간 부담이 커지니까 필요한 만큼만 두는 게 중요하고, 커버링 인덱스처럼 쿼리에 필요한 컬럼이 인덱스만으로 해결되면 테이블 접근을 줄일 수 있습니다. 복합 인덱스는 선행 컬럼부터 실제 조건 순서에 맞게 설계합니다.

**Q. N+1 문제는 뭔가요?**

**A.** 한 번에 목록을 가져온 뒤 루프마다 관련 데이터를 또 쿼리해서 1+N번 나가는 패턴입니다. JOIN이나 IN으로 한 번에 가져오거나 배치 로딩으로 묶고, GraphQL이면 DataLoader 같은 걸 쓰는 식으로 줄입니다.

**Q. 풀 스캔이랑 인덱스 스캔은 어떻게 판단하나요?**

**A.** 데이터량이나 선택도에 따라 달라져서 말로만 단정하기 어렵고, `EXPLAIN`이나 DB에 맞는 실행 계획 도구로 실제 플랜을 보는 게 맞습니다.

## 4.3 ACID, 트랜잭션 경계, 격리 수준, 락

**Q. ACID랑 트랜잭션 경계는 어디에 두시나요?**

**A.** ACID는 원자성·일관성·격리성·지속성이고, 경계는 보통 한 유스케이스가 비즈니스적으로 한 덩어리로 맞물리는 단위에 둡니다. 구현에서는 서비스나 리포지토리 레이어에서 `@Transactional` 같은 걸로 묶는 식으로 씁니다.

**Q. 격리 수준이 뭔가요?**

**A.** Read Committed, Repeatable Read처럼 단계가 있고, 더티 리드나 논리적 팬텀을 얼마나 허용하느냐가 달라집니다. DB마다 기본 격리 수준이 다르고, 팬텀이나 락 대기로 트러블슈팅할 때는 격리 수준과 락을 같이 봅니다.

**Q. 낙관적 락·비관적 락·데드락은요?**

**A.** 낙관적 락은 버전 컬럼으로 나중에 충돌을 검사하는 방식이고, 비관적 락은 `SELECT … FOR UPDATE`처럼 미리 잠그는 방식입니다. 데드락은 서로 다른 순서로 락을 잡을 때 나는데, 락 순서를 통일하거나 타임아웃·재시도로 완화합니다.

## 4.4 커넥션 풀, 마이그레이션, 샤딩·파티셔닝·리플리케이션, DB 특징

**Q. 커넥션 풀은 왜 쓰나요?**

**A.** 매번 TCP 연결이나 인증 비용이 크니까 연결을 재사용하려고 씁니다. 풀 크기는 동시 요청이랑 DB `max_connections`에 맞춰 조정합니다.

**Q. DB 스키마 변경은 어떻게 관리하시나요?**

**A.** Flyway나 Liquibase처럼 마이그레이션 도구로 버전을 남기고, 같은 스키마를 재현 가능하게 관리합니다.

**Q. 파티셔닝, 샤딩, 리플리케이션 차이는요?**

**A.** 파티셔닝은 한 테이블을 구간이나 해시 등으로 나눠 관리하는 거고, 샤딩은 데이터를 여러 노드에 분산하는 거, 리플리케이션은 복제본으로 읽기 부하를 나누거나 고가용성을 노리는 쪽입니다. 더 깊게 물으면 다뤄 본 범위 안에서만 예를 듭니다.

**Q. 사용해 본 DB의 특징을 짧게 말해 보세요.**

**A.** (예시) PostgreSQL을 썼다면 MVCC로 읽기·쓰기가 겹칠 때를 어떻게 다루는지, JSONB로 반구조화 데이터를 다룬 경험을 짚고, MySQL이면 InnoDB 트랜잭션·클러스터드 인덱스를 한두 문장으로 정리합니다.
