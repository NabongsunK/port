## 16. 추가 기술 질문 (렌더링·생명주기·API·서버)

### “브라우저를 렌더링 할 때 어떠한 과정을 거치나요?”

**답변:**  
브라우저는 HTML을 파싱해 DOM 트리를 만들고, CSS를 파싱해 CSSOM을 만든 뒤 두 트리를 합쳐 Render Tree를 구성합니다. 그 다음 요소의 크기와 위치를 계산하는 Layout(Reflow), 실제 픽셀을 그리는 Paint, 레이어를 합성하는 Composite 순서로 화면을 완성합니다.  
JavaScript가 DOM/CSS를 변경하면 Reflow/Repaint가 다시 발생할 수 있어, 성능 최적화 시에는 변경 범위를 줄이거나 `transform`/`opacity` 중심으로 애니메이션을 구성합니다.

### “생명주기를 어떻게 이용을 했나요?”

**답변:**  
React 기준으로 마운트 시 초기 데이터 로딩, 업데이트 시 의존성 변화에 따른 재동기화, 언마운트 시 정리(clean-up) 패턴을 사용합니다.  
예를 들어 `useEffect(() => {...}, [])`로 최초 1회 로직을 실행하고, `useEffect(() => {...}, [dep])`로 특정 상태 변경 시 API를 재요청하며, return 함수에서 이벤트 리스너·타이머·요청 중단 처리를 해 메모리 누수와 중복 호출을 방지합니다.

### “API를 연결할 때 각 요소들은 어떤 순서로 연결되어 있는가?”

**답변:**  
일반적으로 **클라이언트 이벤트 → 요청 생성(토큰/헤더 포함) → 서버 라우터/미들웨어(인증·검증) → 서비스 로직 → DB/외부 API → 응답 반환 → 클라이언트 상태 갱신** 순서로 흐릅니다.  
클라이언트에서는 loading/success/error 상태를 분리해 사용자 경험을 안정화하고, 서버에서는 입력 검증·인가·에러 코드를 명확히 나눠 프론트가 복구 가능한 응답 계약을 유지하는 것을 중요하게 봅니다.

### “정적 서버와 동적 서버를 각각 설명하시오”

**답변:**  
정적 서버는 빌드된 HTML/CSS/JS/이미지 같은 파일을 그대로 제공하는 방식으로, 캐싱과 CDN 활용이 쉬워 응답 속도와 비용 측면에서 유리합니다. 대신 사용자별 개인화나 실시간 계산이 필요한 기능에는 한계가 있습니다.  
동적 서버는 요청 시점에 서버 로직을 실행해 응답을 생성하므로 인증·권한·개인화·실시간 처리에 적합하지만, 서버 부하와 운영 복잡도가 상대적으로 높아집니다.  
실무에서는 한쪽만 고집하기보다 콘텐츠 페이지는 정적(SSG/ISR), 사용자 액션은 동적으로 분리한 하이브리드 구성을 주로 사용합니다.

### “SP(Stored Procedure) 왜 쓰는 걸까?”

- **답변:**  
  SP는 DB에 저장된 실행 로직이라, 복잡한 조회/처리를 애플리케이션 코드 밖에서 일관되게 재사용할 수 있습니다.  
  장점은 **성능(네트워크 왕복 감소, 실행 계획 재사용 가능)**, **보안(테이블 직접 권한 대신 프로시저 실행 권한 부여)**, **업무 규칙 일원화**입니다.  
  다만 로직이 DB에 과도하게 몰리면 테스트/버전관리/이식성이 어려워질 수 있어, 핵심 트랜잭션·집계처럼 DB가 잘하는 영역에 한정해서 쓰는 편이 좋습니다.

### “LEFT JOIN 알고 있냐?”

- **답변:**  
  `LEFT JOIN`은 왼쪽 테이블 기준으로 모든 행을 유지하고, 오른쪽에 매칭되는 값이 없으면 `NULL`로 채워서 가져오는 조인입니다.  
  즉 “기준 목록은 전부 보여주되, 연결 데이터가 있으면 붙인다”는 상황에 적합합니다.

- **예시 SQL:**  
  `users`는 모두 보여주고, 주문이 있는 경우만 `orders`를 붙이는 경우

### 트랜잭션 써보셧나요?

원자성과 데이터 정합성을 지키기 위해 한번에 UPDATE나 INSERT가 열어번 나갈때, 중간에 실패할때 전부 취소해야할떄 결재상태 변경같이 정합성이 깨지면 안될때 사용했습니다.

### nextjs 쓴이유

“포트폴리오는 콘텐츠 중심이라 데이터 변동이 크지 않아 빌드 기반 정적 생성(SSG)으로 운영 비용과 복잡도를 줄이는 게 목표였고, 라우팅·콘텐츠 파이프라인까지 한 번에 잡기 좋아서 Next.js를 선택했습니다. 필요한 경우 같은 프로젝트에서 API 레이어까지 묶어 관리할 수 있어 유지보수에도 유리했습니다.”
