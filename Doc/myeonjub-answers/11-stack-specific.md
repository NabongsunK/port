# 11. 본인 스택에 맞춰 추가 (예시)

**전부 외울 필요 없음.** 이력서에 적은 스택만 골라 깊이를 맞춥니다.

---

## Java / Spring

**답변 요지:**

- **빈 스코프**: singleton(기본) vs prototype, 웹 요청 단위 request scope 등.
- **AOP**: 횡단 관심(로깅, 트랜잭션)을 프록시로 분리.
- **트랜잭션 전파**: `REQUIRED`, `REQUIRES_NEW` 등 시나리오별 선택.
- **JPA**: N+1(`@EntityGraph`, fetch join), 영속성 컨텍스트·더티 체킹 개념.

---

## Node.js

**답변 요지:**

- **cluster**: 여러 워커 프로세스로 포트 공유(구식 패턴, 오늘은 PM2·컨테이너 스케일이 흔함).
- **worker_threads**: CPU 바운드 작업을 메인 이벤트 루프와 분리.
- **PM2**: 프로세스 관리·재시작·로그 — 운영 경험이 있으면 한 줄.

---

## Python

**답변 요지:**

- **GIL**: CPU 바운드 멀티스레드 한계, I/O 위주는 괜찮은 경우가 많음.
- **FastAPI/Django**: 비동기(`async def`) vs 동기 WSGI 맥락을 본인 사용에 맞게.

---

## gRPC·Protocol Buffers

**답변 요지:**

- 바이너리 직렬화로 **작은 페이로드·스키마 진화**(필드 번호) 관리.
- **스트리밍**: 단방향·양방향 RPC로 실시간·대량 데이터 전송.

### “전자결재를 구현하셨네요? 어떻게 구현하셨어요?”

전자결재를 구현할 때는 먼저 “결재의 흐름”을 상태로 모델링했습니다. 현재 무슨 단계인지에 따라, 다음으로 허용되는 액션이 달라지도록 정의했습니다. 그래서 화면에서 버튼이 보이는 것만이 아니라, 서버에서도 현재 상태가 승인/반려 처리 가능한 단계인지 먼저 검증하고 상태 전이를 수행했습니다.

그 다음으로 권한 분리를 확실히 했습니다. 관리자와 피관리자는 “누구인지”가 다르고, 그에 따라 볼 수 있는 리스트와 검색 범위가 달라야 했기 때문에 조회 API에서 서버 조건을 역할별로 분기했습니다. 예를 들어 피관리자는 본인이 올려야 할 문서 위주로 조회되게 하고, 관리자는 결재 처리 대상이 되는 문서만 조회되게 했습니다. 승인/반려 같은 “처리 액션”도 마찬가지로, 해당 문서의 현재 단계에서 그 역할이 처리할 권한이 있는지 서버가 검증한 뒤에만 업데이트가 되게 구현했습니다.

DB 쪽은 결재 처리 시 데이터 무결성이 깨지지 않도록 트랜잭션으로 묶었습니다. 결재 처리와 상태 변경, 결재 이력 저장 같은 작업을 한 번에 원자적으로 처리해서 “승인했는데 이력은 안 남는” 같은 불일치를 방지했고, 필요하면 Stored Procedure로 로직을 집중시켜 처리 성능과 일관성을 챙겼습니다. 결과적으로 클라이언트가 어떤 값을 보내든 서버에서는 “문서 상태 + 역할 권한”을 기준으로만 업데이트가 일어나게 했기 때문에 안정적으로 운영할 수 있었습니다.

마지막으로, 이런 방식의 장점은 유지보수성입니다. 워크플로 상태와 권한을 기준으로 분기하다 보니 단계가 늘어나도 서버 규칙이 명확하게 확장되고, 장애가 나면 “어느 단계에서 어떤 권한 검증이 실패했는지”를 로그/이력으로 추적하기 쉬웠습니다.

#### (꼬리질문) “인증/인가 차이”

인증은 “요청한 사용자가 누구인지” 확인하는 단계입니다. 저희는 **서버 세션** 으로 로그인 상태를 유지했고, 요청마다 세션이 유효한지·어떤 사용자인지 먼저 확인했습니다. 인가는 그다음 단계로, “그 사용자가 **지금 이 문서·이 단계**에서 조회·승인·반려 같은 행동을 해도 되는지”를 기준으로 검증하는 겁니다.

전자결재에서는 인증만 통과했다고 끝이 아니라, 목록·검색은 **관리자/피관리자에 따라 조회 범위를 다르게** 두었고, 승인·반려는 **문서 상태가 처리 가능한 단계인지**와 **그 역할이 그 단계를 처리할 권한이 있는지**를 서버에서 리소스 단위로 다시 봤습니다. JWT를 쓰지 않았다고 인가가 약해지는 건 아니고, **세션으로 누구인지 정한 뒤 인가 로직을 거는 구조**는 흔한 패턴입니다.
