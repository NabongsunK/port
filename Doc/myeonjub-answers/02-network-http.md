# 2. 네트워크·HTTP (서버 관점)

## URL 입력 후 서버까지 흐름 (DNS, TCP, TLS)

**답변 요지:**

1. 브라우저가 도메인을 IP로 바꾸기 위해 **DNS 조회**
2. 서버와 **TCP 3-way handshake**로 연결
3. HTTPS면 **TLS 핸드셰이크**로 암호화 협상·인증서 검증
4. **HTTP 요청**(메서드, 경로, 헤더, 바디) 전송
5. 서버가 처리 후 **HTTP 응답**  
   백엔드 면접에서는 “우리 API는 TLS 종료가 Nginx에서 되고, 뒤로 프록시한다”처럼 **본인 환경**을 한 줄 넣으면 좋습니다.

---

## HTTP vs HTTPS, 상태 코드 4xx·5xx

**답변 요지:**  
HTTPS는 HTTP 위에 TLS를 얹어 내용을 암호화하고 서버 신원을 검증합니다.

- **4xx**: 클라이언트 쪽 문제(인증·권한·잘못된 요청). 재시도해도 같은 요청이면 같은 결과인 경우가 많음.
- **5xx**: 서버·게이트웨이 오류. 일시적일 수 있어 재시도·알림 대상.  
  로그와 모니터링으로 5xx 비율을 보고, 원인은 애플리케이션 로그·DB·다운스트림 API로 좁힙니다.

---

## HTTP 메서드와 멱등성

**답변 요지:**

- **GET**: 조회, **멱등**(여러 번 해도 결과 자원 상태가 바뀌지 않게 설계)
- **POST**: 생성 등, 비멱등
- **PUT**: 전체 치환, 멱등으로 두는 경우가 많음
- **PATCH**: 부분 수정, 스펙에 따라 멱등이 아닐 수 있음
- **DELETE**: 삭제, 멱등으로 취급하는 경우가 많음  
  **멱등**은 “같은 요청을 실수로 두 번 보내도 안전한가”에 가깝게 이해하면 됩니다.

---

## 헤더: `Content-Type`, 인증, `Cache-Control`

**답변 요지:**

- `Content-Type`: 바디 형식(JSON, form-data 등). API는 보통 `application/json`.
- 인증: `Authorization: Bearer <JWT>` 등. 서버는 미들웨어/필터에서 검증.
- `Cache-Control`: 브라우저·CDN 캐시 동작. API 응답은 `no-store`로 두는 경우도 많고, GET 조회 API는 짧은 `max-age`를 줄 수도 있습니다.

---

## CORS가 생기는 이유, 서버 처리

**답변 요지:**  
브라우저가 **다른 출처(origin)** 로 요청할 때 보안상 응답을 막고, 서버가 `Access-Control-Allow-Origin` 등으로 허용을 명시해야 합니다.  
백엔드에서는 허용할 오리진·메서드·헤드를 설정하고, **프리플라이트(OPTIONS)** 요청에도 맞춰 응답합니다. 서버 간 호출(curl, 다른 백엔드)에는 CORS가 없습니다.

---

## 쿠키·세션·토큰 검증·발급

**답변 요지:**

- **세션**: 서버(또는 Redis)에 세션 ID로 상태 저장, 쿠키에는 세션 ID만.
- **JWT**: 서버가 서명한 토큰을 클라이언트가 들고 다니고, 서버는 서명·만료·클레임을 검증.
- 로그아웃·강제 만료가 필요하면 블랙리스트·짧은 만료+리프레시 토큰 전략을 씁니다.  
  HttpOnly·Secure 쿠키로 XSS로 토큰 탈취 위험을 줄입니다.
