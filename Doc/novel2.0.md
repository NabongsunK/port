이제는 소수의 전문가만이 아니라 누구나 자신의 콘텐츠를 만들고 배포하는 크리에이터의 시대라고 생각합니다. 그만큼 도구의 사용성은 단순한 편의 기능을 넘어 창작의 속도와 완성도를 결정하는 핵심 요소가 되었다고 봅니다. 저는 프론트엔드 개발자로 일하며 사용자가 더 빠르고 직관적으로 결과물을 만들 수 있도록 화면 흐름, 인터랙션, 성능을 꾸준히 개선해 왔고, 이러한 경험을 툴디의 웹 디자인 편집기 고도화에 연결하고 싶어 지원했습니다.

"지원 동기"
저는 퍼즐과 수학 문제를 통해 구조적으로 사고하는 습관을 길렀고, 수학과 진학 이후에는 문제를 논리적으로 분해해 해결하는 과정에 흥미를 느꼈습니다. 대학 시절 알파고와 이세돌의 대결을 계기로 프로그래밍에 매료되었고, 컴퓨터 동아리 활동에서 C++로 문제를 해결하면서 코드로 현실의 문제를 푸는 일의 매력을 확신하게 되었습니다. 이후에는 기술 자체보다 사용자가 실제로 체감하는 가치가 무엇인지 먼저 고민하는 개발자로 성장해 왔습니다.

"프론트엔드를 선택하게 된 계기"
프론트엔드를 선택한 계기는 제가 좋아하던 게임 데이터를 더 쉽고 빠르게 보여주고 싶다는 호기심이었습니다. 사용자가 원하는 정보를 직관적으로 확인할 수 있는 화면을 직접 만들고 싶어 JavaScript와 React를 본격적으로 학습했고, 멀티캠퍼스 프론트엔드 과정에서 팀 프로젝트를 수행하며 실전 감각을 키웠습니다.

교육 수료 후에는 학습 내용을 실제 서비스로 확장하고자 팀을 구성해 ALOA 프로젝트를 진행했습니다. 스마일 게이트에서 제공하는 API를 활용하여 사람들이 자신의 게임 캐릭터를 한눈에 확인하고, 이를 통해 파티를 구성하거나 게임을 함께할 수 있는 작은 연결을 제공하는 서비스를 기획했습니다. 기획부터 UI 설계, API 연동, 상태 관리까지 전 과정을 경험했습니다. 특히 배포 이후 사용자 피드백을 반영해 기능을 개선하는 과정을 반복하며 개발은 구현으로 끝나는 일이 아니라 운영과 개선까지 포함된 일이라는 점을 체감했습니다.

"나의 강/단점"
실무에서는 화재보험협회 내부 시스템을 VB.NET 기반 레거시에서 Vue.js와 Go 기반 구조로 리팩토링하는 업무를 맡았습니다. 당시 세 기술 모두 익숙하지 않았지만, 낯선 기술을 빠르게 습득하는 학습 민첩성과 복잡한 코드를 패턴화해 표준으로 정리하는 구조화 능력을 강점으로 삼아 대응했습니다. 먼저 기존 코드의 반복 패턴을 분류하고 공통 변환 규칙을 정의해 작업 단위를 표준화했으며, 필요한 부분에서는 AI 도구를 보조적으로 활용해 학습 시간을 줄였습니다. 그 결과 리팩토링 속도와 산출물의 일관성을 함께 높일 수 있었습니다.

저의 강점은 일정 압박과 기술적 변수가 겹친 위기 상황에서도 흔들리지 않고 목표를 완수하는 책임감입니다. 실제 프로젝트 마감 직전, 예상치 못한 이슈가 연달아 발생하며 배포 일정에 차질이 생길 위기에 처한 적이 있습니다. 저는 무조건적인 작업 연장보다는 효율적인 상황 판단이 우선이라 판단했습니다. 먼저 전체 기능을 '필수'와 '후순위'로 재분류한 뒤, 실무진과 긴급 협의를 거쳐 단계적 마감 기준을 수립했습니다. 이를 통해 리소스를 핵심 기능에 집중시켜 정해진 기한 내에 성공적으로 배포를 마칠 수 있었습니다.

배포 이후에도 책임감 있는 자세를 견지했습니다. 안정적인 운영을 위해 실무진의 모니터링 요청이 있었고, 저는 프론트엔드와 백엔드를 모두 아우르는 역량을 바탕으로 자원하여 당일 업무 종료 시점까지 이상 징후를 면밀히 살폈습니다. 이후 단순한 문제 해결에 그치지 않고 이를 팀의 자산으로 만들고자 노력했습니다. 대응 절차를 매뉴얼화하여 팀 내 공유함으로써 운영 효율을 높였고, 일정 관리 단계를 정교화해 예측 가능성을 확보했습니다. 위기 상황에서 동료들이 믿고 의지할 수 있는 '해결사'이자, 지속 가능한 개발 환경을 고민하는 개발자로 거듭나겠습니다.

"경험 및 경력"
(주)가이랩에서 2024년 4월부터 2025년 12월까지 웹·서버 개발 업무를 수행했습니다. 화재보험협회 내부 업무 시스템을 VB.NET 기반 레거시를 웹(SPA)·REST API·DB 계층 구조로 전환·고도화하는 프로젝트에 참여했습니다. 프론트엔드는 React·Vue, 백엔드는 Golang, FastApi(Python), DB는 MSSQL, MongoDB, 추가로, Figma, Docker 등을 업무 범위에 따라 병행했습니다.

첫째, WinForms·프로시저·뷰를 분석·정리하고 Golang 기반 REST API를 구현했습니다. 기존 이벤트 기반 코드에 엮여있는 쿼리와 데이터 접근을 Controller–Service–Repository 구조로 옮기고, 폼 이벤트에 섞인 로직을 화면·도메인·데이터 접근으로 나누어 쉽게 유지보수 가능한 형태로 재설계했습니다. 팀원 교체에 대비해 레거시 분석 결과·API 명세·배포·실행 방법을 온보딩 문서로 남겼고, 일정이 촉박할 때는 담당자와 소통하며 필수 기능과 부가 기능을 구분해 마감 기준을 조정하며 프로젝트를 마무리했습니다.

같은 기간에 점검 계획 기능을 구현하는 프로젝트를 맡았습니다. 실무에서 정한 점검·항목·담당자·이행 여부를 시스템에 담기 위해 화면과 REST API, DB 스키마를 설계·구현했고, 점검 등록·조회·상태 변경이 한꺼번에 일어나는 흐름에서 동시성과 데이터 정합성 이슈가 커질 수 있어, 트랜잭션의 커밋·롤백 경계를 명확히 하고 한 번에 반영하는 배포·변경 단위를 나누어, 문제 발생 시 롤백과 재배포로 영향 범위를 통제할 수 있도록 했습니다. 점검 결과와 이상 징후 대응까지 이어지는 시나리오를 끊기지 않게 하기 위해, 주요 API·화면·로그 확인 순서를 체크리스트로 정리해 구간마다 테스트 할 수 있게 했습니다.

이후엔 1차 전환으로 구축한 Web+API 구조를 유지한 채 전자결재와, MD 산정 기능을 확장했습니다. 결재 단계·상태·권한을 반영한 Workflow와 Stored Procedure 기반 결재 트랜잭션을 설계·구현했고, MD 산정은 T-SQL로 계산·집계·조회 로직을 정형화한 뒤 실적 관리·조회 웹 화면까지 연결했습니다. 보안 점검 대응 과정에서는 입력 검증·권한·감사 로그를 공통 미들웨어 수준에서 보완했으며, 정책 변경이 잦은 영역은 설정·코드 테이블 등으로 분리해 코드 수정 없이 조정할 수 있도록 했습니다.

마지막으로 Python으로 작성된 데이터 처리 로직을 FastAPI 기반 API 서비스로 재구성하고, 스케줄러를 통해 정기 배치를 자동화했습니다. Hugging Face·LangChain을 활용한 임베딩·검색·질의응답 파이프라인을 설계·구현했으며, 스크립트 가정에서 벗어나 구간 단위 재시도·로깅·실행 결과 코드를 남기는 등 서비스 운영에 필요한 책임 범위를 정리했습니다. 배치와 온라인 서비스의 피크 타임·부하를 고려해 실행 시각·청크 처리·레이트 리밋 등 운영 기준을 잡았고, 기존 서비스와의 연동에서는 API 계약·버전·문서화와 검증 절차를 정리해 변경 비용을 줄이는 데 주력했습니다.

위 경험을 바탕으로 레거시 이해와 신규 도메인 확장, API·배치 운영까지 폭넓게 다루어 왔으며, 툴디의 웹 디자인 편집기·프론트엔드 고도화 업무에서도 동일한 기준으로 사용자 경험과 안정적인 릴리스에 기여하고자 합니다.
