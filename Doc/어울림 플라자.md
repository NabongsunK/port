어울림플라자에 지원한 이유는 무엇입니까?
어울림플라자가 배리어프리(barrier-free) 기반의 포용적 공간을 만들고 시민의 문화·복지 접근성을 높인다는 점에 깊이 공감했습니다. 배리어프리는 장애인·고령자 등 사회적 약자에게 불편한 물리적·심리적·제도적 장벽을 허물어, 사회 구성원 모두가 함께 살기 좋은 환경을 만들자는 의미라고 이해하고 있습니다. 저는 서비스가 ‘열려 있는 상태’로 안정적으로 운영되도록 API·DB·배치·운영 관점에서 품질을 설계해 왔고, 어울림플라자의 운영 품질을 시스템적으로 뒷받침하는 역할로 기여하고 싶어 지원했습니다.

어울림플라자에서 이루고 싶은 꿈은 무엇입니까?
시설 운영은 물론, 이용자 경험을 좌우하는 디지털 운영(예약/프로그램/안내 등)에서도 신뢰성과 예측 가능성을 높이고 싶습니다. 점검과 배포, 이상 징후 대응을 표준화해 운영 리스크를 낮추고, 데이터 기반으로 프로그램 품질과 접근성(대기/오류/지연)을 지속 개선하는 “운영이 곧 서비스”가 되게 만드는 것이 목표입니다.

지원부서를 선택한 이유와 본인이 갖춘 역량은 무엇입니까?
제가 선택한 직무는 시민이 참여·연수를 신청하고 이용하는 디지털 채널은 한 번의 오류·지연도 신뢰로 이어질 수 있어, “기능을 만드는 것”을 넘어 운영 안정성을 핵심으로 두고, 아래와 같은 우대 역량과 실무 기반 기술을 갖추고 있습니다.

[멀티캠퍼스] 프론트엔드 개발자 교육과정 에서 JavaScript와 React를 학습을 했고, 팀장으로써 React/NodeJs로 이루어진 프로젝트로 최우수상을 수상하였습니다. 해당 실습이 종료된후, 실제 프로젝트 경험을 해보고 싶어서 팀원들을 다시모아 ALOA 프로젝트를 실시하였습니다. 이 프로젝트에서는 스마일 게이트에서 제공하는 API를 활용하여 사람들이 자신의 게임 캐릭터를 한눈에 확인하고, 이를 통해 파티를 구성하거나 게임을 함께할 수 있는 작은 연결을 제공하는 서비스를 기획했고, React와 Nodejs로 이루어진 결과물은 AWS EC2를 사용하여 배포하였고, 사용자 정보같은 데이터는 AWS RDS에 띄운 MySQL에 저장하였습니다.

실무에서는 화재보험협회 내부 업무 시스템의 레거시(VB.NET WinForms) 전환·고도화에 참여하며 React·Vue, Golang·FastAPI(Python), MSSQL·MongoDB를 함께 다뤘습니다. REST API 설계·구축, Stored Procedure 기반 데이터 처리, 트랜잭션 경계 설계, GitLab 협업을 통해 운영 중 변경 리스크를 줄이는 방식으로 업무를 수행했습니다. 또한 모니터링·체크리스트·매뉴얼로 점검-대응 절차를 표준화해 운영 효율을 높였습니다.

직장 생활에서 드러내고 싶은 경험이나 성과는 무엇입니까?
직전 회사에서는 직원들이 업무 시간에 실시간으로 사용하는 시스템을 다뤘기 때문에, 기능 구현 속도와 배포 안정성을 동시에 확보하는 것이 핵심이었습니다. 저는 유닛 테스트와 체크리스트 기반 검증 절차를 적용하고, 롤백 가능한 배포 단위를 기준으로 변경을 관리해 운영 중 장애 리스크를 줄였습니다. 특히 마감 직전 변경이 몰리는 시기에도 필수/후순위 기능을 구분해 반영 순서를 조정하며 일정과 안정성을 함께 맞췄습니다.

첫번째 했던 프로젝트에서 레거시(VB.NET WinForms) 기반 시스템을 웹(SPA)+REST API 구조로 전환하며 화면·비즈니스 로직·데이터 접근을 분리해 유지보수성을 높였습니다. 이 과정에서 장애 발생 시 원인 파악과 수정까지 걸리는 시간을 평균 1일에서 반나절 수준으로 단축하는 기준을 만들고, 배포 전후 점검 항목을 체크리스트와 문서로 정리해 팀이 동일한 품질 기준으로 검증할 수 있게 했습니다.

같은 기간 점검 계획 기능에서는 점검 등록·조회·상태 변경이 연속으로 발생하는 흐름에서 동시성과 데이터 정합성이 깨지지 않도록 커밋/롤백 경계를 명확히 하고, 배포·변경 단위를 분리해 영향 범위를 통제했습니다. 점검 결과와 이상 징후 대응까지 이어지는 시나리오를 체크리스트로 묶어, 대응 단계에서의 재현·확인 시간을 줄였습니다.

이어 전자결재·MD 산정 확장 단계에서는 결재 Workflow의 단계·상태 전이를 반영하고 Stored Procedure 기반 트랜잭션으로 처리 일관성을 확보했습니다. 또한 권한 검증·입력 검증·감사 로그를 공통 미들웨어 수준에서 보완해 보안 점검 지적사항의 재발 가능성을 낮췄고, MD 산정 기준을 시스템화하면서 사용자 요청사항을 주 50건 수준에서 월 5건 이하로 감소시키는 성과를 만들었습니다.

마지막으로 FastAPI 기반 데이터 처리/배치 서비스에서는 Python 로직을 서비스 수준으로 전환하며 구간 단위 재시도·로깅·실행 결과 코드를 설계해 운영 부담을 줄였습니다. 피크 타임에는 청크 처리와 레이트 리밋 기준을 두어 API와 배치가 서로 방해하지 않도록 운영 품질을 확보했습니다. 이 경험을 통해 저는 “개발 완료”가 아니라 “안정적으로 운영되는 상태”를 성과의 기준으로 삼게 됐습니다.

본인은 어떤 사람인지 기술하여 주시기 바랍니다.
저는 책임감을 바탕으로 마감과 운영을 끝까지 완결하는 사람입니다. 기술적으로는 복잡한 로직을 패턴화해 표준으로 정리하는 편이고, 일정 압박 상황에서도 우선순위를 재정의해 흔들리지 않게 마무리해 왔습니다. 문제 해결 이후에는 같은 상황이 반복되지 않도록 절차를 문서화하고 팀이 바로 따라갈 수 있게 공유하는 습관이 있습니다. 협업에서는 상대의 의도를 신뢰하고, 필요한 합의를 문서와 체크리스트로 남기는 방식으로 커뮤니케이션합니다.

추가로 작성하고 싶은 사항이 있다면 자유롭게 기재하여 주시기 바랍니다.
공공시설 특성상 이용자에게 보이지 않는 운영 안정성이 더 중요하다고 생각합니다. 입·출력 오류, 데이터 정합성, 권한/보안 이슈가 운영 품질을 흔들지 않도록 점검 체계를 갖추고, 배포/변경 시에는 롤백 가능한 실행 단위와 검증 순서를 표준화하겠습니다. 또한 배리어프리와 포용 가치에 맞춰 디지털 서비스도 “접근 가능한 품질”로 개선하는 관점으로 빠르게 학습해 기여하겠습니다.
